# make rpiboot — Containerized rpiboot for eMMC Flashing

## Requirements

- `make rpiboot` puts the reTerminal's CM4 eMMC into USB mass storage mode so the host can `dd` an image to it.
- Zero manual `git clone`, `brew install`, or `make` steps. One command.
- Pin the usbboot commit for reproducibility.
- Works on macOS (Apple Silicon). Linux is a future nice-to-have.

## Detailed Implementation Plan + Reasoning

### Why containerization doesn't work

rpiboot talks to the CM4's BootROM over raw USB (libusb). On macOS, containers run inside a Linux VM (Podman applehv / Docker LinuxKit). USB devices on the host are **not visible** inside the VM. The available workarounds are all broken or impractical:

| Approach | Status |
|---|---|
| Docker USB/IP (4.35.0+) | No mature macOS USB/IP server exists. |
| Podman `--usb` (QEMU) | Only works with QEMU provider, which is broken on Apple Silicon for USB passthrough. |
| Podman `--usb` (applehv) | Not supported at all. |
| VirtualHere (commercial) | Too much setup friction for a dev tool. |

**Conclusion: rpiboot must run natively on the macOS host.** This is also the upstream-recommended path — the usbboot repo explicitly supports macOS.

### Approach: `make rpiboot` builds and runs rpiboot natively

Instead of containerizing, we automate the clone-build-run cycle:

1. Clone `raspberrypi/usbboot` (pinned commit, with submodules) into `build/usbboot/`.
2. Build the `rpiboot` binary natively using the host's clang + Homebrew libusb.
3. Run `sudo rpiboot -d mass-storage-gadget64` to put the CM4 eMMC into mass storage mode.

This is fully automated — the user just runs `make rpiboot`.

### Design decisions

- **Clone into `build/usbboot/`** — keeps it out of the repo root, already under `build/` which is gitignored. `make clean` should remove it.
- **Pin by commit, not tag** — usbboot doesn't do releases/tags. Pin the commit hash in the Makefile for reproducibility.
- **`--shallow-submodules`** — the rpi-eeprom submodule is huge; shallow submodules keep the clone small. The top-level repo is a full clone so we can checkout the pinned commit.
- **Prereqs: `libusb` and `pkg-config` via Homebrew** — the Makefile should check for these and print a clear error if missing, not silently fail. Don't auto-install Homebrew packages.
- **`sudo` is required** — rpiboot needs root for raw USB access on macOS. The Makefile target runs `sudo` explicitly. This is consistent with the upstream docs.
- **`-d mass-storage-gadget64`** — the fast path for CM4 (BCM2711). Ships firmware over USB, CM4 re-enumerates eMMC as mass storage. This is the current recommended mode per the flash catalog.
- **Separate `make rpiboot` from `make flash`** — rpiboot only puts the eMMC into mass storage mode. The actual `dd` is a separate step (or a future `make flash` target). Reason: the user needs to identify the correct `/dev/diskN` before writing, and auto-detecting it is error-prone on macOS.

### Makefile additions

```makefile
# --- rpiboot (native, not containerized) ---
USBBOOT_REPO   := https://github.com/raspberrypi/usbboot
USBBOOT_COMMIT := 42ca50932f67f4571951a11da3c3161561cb49c2
USBBOOT_DIR    := build/usbboot
RPIBOOT        := $(USBBOOT_DIR)/rpiboot

$(USBBOOT_DIR)/.git:
	git clone --recurse-submodules --shallow-submodules \
		$(USBBOOT_REPO) $(USBBOOT_DIR)
	cd $(USBBOOT_DIR) && git checkout $(USBBOOT_COMMIT) && git submodule update --recursive

$(RPIBOOT): $(USBBOOT_DIR)/.git
	@command -v pkg-config >/dev/null || { echo "ERROR: pkg-config not found. Run: brew install pkg-config"; exit 1; }
	@pkg-config --exists libusb-1.0 || { echo "ERROR: libusb not found. Run: brew install libusb"; exit 1; }
	$(MAKE) -C $(USBBOOT_DIR)

.PHONY: rpiboot
rpiboot: $(RPIBOOT)
	sudo $(RPIBOOT) -d $(USBBOOT_DIR)/mass-storage-gadget64
```

### gitignore

`build/usbboot/` is already covered by `build/` patterns. No changes needed.

### `make clean` update

Add `$(USBBOOT_DIR)` to the `rm -rf` in the clean target (or it's already covered by `build/repos` — no, `build/usbboot` is separate). Add it.

### Future: `make flash`

Out of scope for this plan but the natural next step. Would:
1. Auto-detect the eMMC disk (look for a new ~29GB external disk).
2. Decompress + `dd` the artifact to it.
3. Eject.

Risky on macOS due to disk misidentification. Safer to keep it manual or require the user to confirm the device.
