# Ship docs/ as On-Device Manpages

## Requirements

1. All `docs/*.md` files are converted to manpages and installed on the device image.
2. Users can read them with `man appliance-building`, `man appliance-updates`, etc.
3. The conversion happens at build time with no new layer dependencies (no `meta-virtualization`, no `go-md2man`, no `pandoc`).
4. The README documents the self-replicating design goal: everything needed to understand, rebuild, reflash, and maintain the appliance is shipped on the device itself.
5. `man-db` is installed so `man` works on the device.

## Detailed Implementation Plan + Reasoning

### Manpage naming convention

Each doc becomes a section 7 manpage (miscellaneous/overview documentation):

| Source file | Manpage name |
|---|---|
| `docs/building.md` | `appliance-building(7)` |
| `docs/flashing.md` | `appliance-flashing(7)` |
| `docs/post-install.md` | `appliance-post-install(7)` |
| `docs/updates.md` | `appliance-updates(7)` |
| `docs/dependencies.md` | `appliance-dependencies(7)` |
| `docs/layers.md` | `appliance-layers(7)` |
| `docs/reterminal-quirks.md` | `appliance-reterminal-quirks(7)` |
| `README.md` | `appliance(7)` |

Section 7 is correct for project documentation that isn't a command reference (section 1) or system config (section 5). The `appliance-` prefix namespaces them cleanly.

`docs/TODO-flash-workflow.md` is excluded — it's a development planning doc, not user-facing documentation.

### Markdown-to-troff conversion: no external tools

Adding `go-md2man` would require pulling in `meta-virtualization` or vendoring the recipe. `pandoc` doesn't exist in any upstream layer. Both are overkill for our docs, which use a simple subset of Markdown (headings, paragraphs, code blocks, tables, lists, inline code).

Instead, write a Python script (`md2man.py`) that runs at `do_compile` time. Python 3 is always available on the Yocto build host. The script needs to handle:

- `#` headings → `.SH` (section heading)
- `##`/`###` headings → `.SS` (subsection heading)
- Code blocks (```) → `.nf` / `.fi` (no-fill)
- Bullet lists (`-`/`*`) → `.IP \(bu 2` (indented paragraph with bullet)
- Inline code (`` ` ``) → `\fB...\fR` (bold)
- Bold (`**`) → `\fB...\fR`
- Tables → formatted with `.TS`/`.TE` (tbl) or simplified to plain text
- Links `[text](url)` → `text <url>`
- Paragraphs → `.PP`

Tables are the hardest part. Two options:
1. Use `.TS`/`.TE` (requires `tbl` preprocessor, which groff provides). Clean but adds a dependency on groff-base.
2. Flatten tables to indented text. Simpler, no extra deps.

Option 2 is safer — `man-db` on a minimal image may not have full groff. A simple column-aligned plain text table is readable enough for our purposes.

### Recipe structure

Create a new recipe: `appliance-docs`:

```
layers/meta-appliance-os/recipes-core/appliance-docs/
├── appliance-docs_1.0.bb       # Recipe
├── files/
│   └── md2man.py               # Markdown → troff converter
```

The recipe:
- `SRC_URI` points to the docs directory via `file://` or copies from `APPLIANCE_REPO_ROOT`.
- `do_compile` runs `md2man.py` on each markdown file to produce `.7` manpages.
- `do_install` installs them to `${D}${mandir}/man7/`.
- `RDEPENDS` includes `man-db` so the `man` command is available.

Using `APPLIANCE_REPO_ROOT` (already defined in layer.conf) to find the docs at build time avoids SRC_URI gymnastics. The recipe can glob `${APPLIANCE_REPO_ROOT}/docs/*.md` directly.

### man-db

`man-db` is in poky (`meta/recipes-extended/man-db/`). Add it to IMAGE_INSTALL alongside `appliance-docs`. It provides the `man` command and the `mandb` indexer. The manpage database gets built during `do_rootfs` via an intercept script that runs `mandb`.

Actually, on a read-only rootfs, `mandb` can't update its cache at runtime. Two options:
1. Pre-build the man database during `do_rootfs` (already handled by poky's `rootfs-postcommands` if man-db is installed).
2. Use `man -c` (skip cache) or configure man-db to work without a cache.

Poky's `man-db` package runs `mandb` in a postinst script, which runs during `do_rootfs` under pseudo. This should populate the cache. Since our rootfs is mounted read-only, the cache will be baked in and won't need runtime updates. New manpages only arrive via RAUC updates, which replace the entire rootfs — so the cache is always correct.

### IMAGE_INSTALL wiring

Add `appliance-docs` to the `kiosk:` local_conf_header in `kas/common.yaml` (or create a new `docs:` header). `man-db` should be an RDEPENDS of the recipe, not a separate IMAGE_INSTALL entry, so installing the docs package automatically pulls in the `man` command.

### README self-replicating note

Add a "Design principles" or similar section to README.md that states:
- The appliance is designed to be self-replicating: all documentation needed to rebuild, reflash, update, and maintain the system ships on the device itself as manpages.
- Run `man appliance` on the device for an overview, or `man -k appliance` to list all available documentation.
- Source docs live in `docs/` as Markdown; they are converted to manpages at build time.

### What about the README itself?

The top-level `README.md` becomes `appliance(7)` — the overview manpage. It's the entry point users see when they run `man appliance`. The README's content (hardware table, quickstart, app list) makes sense as a manpage too.
