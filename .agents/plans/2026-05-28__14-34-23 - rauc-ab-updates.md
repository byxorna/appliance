# RAUC A/B Update Wiring Plan

Implements Phase 7 (RAUC A/B updates) and Phase 8 (persistent data) from the master plan. Transforms the current single-rootfs rw image into an immutable A/B dual-rootfs layout with RAUC OTA updates, U-Boot slot selection, and persistent `/data` and `/home` partitions.

## Requirements

1. **A/B rootfs layout**: 5-partition eMMC layout matching the upstream meta-rauc-raspberrypi design: boot (FAT32) + rootfs-A (ext4, ro) + rootfs-B (ext4, ro) + data (ext4, rw) + home (ext4, rw, growfs).
2. **Immutable rootfs**: mounted read-only. Writable overlays via tmpfs for `/tmp`, `/var/volatile`, `/run`.
3. **U-Boot slot selection**: RPi firmware loads U-Boot; U-Boot's `boot.scr` selects A or B based on `BOOT_ORDER` / `BOOT_<X>_LEFT` counters, loads kernel + DTBs from the chosen rootfs partition (not from FAT).
4. **RAUC service**: installed on the target, `system.conf` defines two rootfs slots, uses U-Boot bootloader backend.
5. **RAUC bundles**: `update-bundle.bb` produces a signed `.raucb` containing an ext4 rootfs image. Dev signing keys for now.
6. **Persistent `/data`**: survives updates. Subdirs: `/data/platform/`, `/data/apps/<name>/`.
7. **Persistent `/home`**: survives updates. Grows via `x-systemd.growfs` to fill remaining eMMC. Used for user home directories.
8. **Mark-good**: systemd service that calls `rauc status mark-good` after critical services are up.
9. **Rollback**: if mark-good never fires (boot fails), U-Boot exhausts `BOOT_<X>_LEFT` (3 attempts) and falls back to the other slot.

## Detailed Implementation Plan + Reasoning

### Step 1: Enable U-Boot

U-Boot is required because RAUC's RPi integration uses U-Boot environment variables (`BOOT_ORDER`, `BOOT_A_LEFT`, `BOOT_B_LEFT`) to select the active slot. The RPi firmware loads `u-boot.bin` from FAT instead of loading the kernel directly.

**Changes:**
- Set `RPI_USE_U_BOOT = "1"` in kas `local_conf_header` (not in machine conf, since we don't own the machine conf).
- Set `ENABLE_UART = "1"` — required for U-Boot serial console and already set in the upstream seeed-reterminal.conf.
- The meta-rauc-community layer already provides a `u-boot_%.bbappend` that patches the rpi_arm64_defconfig to enable squashfs support (for reading ext4 rootfs).

**Risk: RPi firmware 16s watchdog.** U-Boot must pet or disable it quickly. The meta-raspberrypi layer's U-Boot integration handles this (the `boot.scr` chain runs within seconds). The AGENTS.md gotcha documents this — monitor during hardware test.

### Step 2: Activate RAUC distro feature

Adding `rauc` to `DISTRO_FEATURES` activates `rauc-integration.bbclass` which adds the `:rauc-integration` override. This enables all the conditional logic in meta-rauc-community's bbappends:
- `rpi-u-boot-scr.bbappend` → installs the A/B `boot.cmd.in`
- `linux-raspberrypi_%.bbappend` → adds `rauc.cfg` kernel config fragment (dm-verity, loop, squashfs, sha256, nbd)
- `core-image-minimal.bbappend` → moves kernel into rootfs, removes it from FAT boot files

**Changes:**
- `DISTRO_FEATURES:append = " rauc"` in `appliance-os.conf` distro conf.
- `IMAGE_INSTALL:append = " rauc"` in kas `local_conf_header` (ensures the RAUC client is in the image).

### Step 3: 5-partition WKS layout

We adopt the upstream meta-rauc-raspberrypi 5-partition layout. Our custom `appliance-dual-rootfs.wks.in` mirrors `sdimage-dual-raspberrypi.wks.in` with minor adjustments (128M boot instead of 100M, 1G data instead of 100M).

**Partition layout:**

| Part | Device | Mount | Type | Size | Notes |
|------|--------|-------|------|------|-------|
| p1 | mmcblk0p1 | `/boot` | vfat | 128M | RPi firmware + U-Boot + config.txt + boot.scr + DTB overlays |
| p2 | mmcblk0p2 | `/` | ext4 | (auto) | rootfs slot A, read-only |
| p3 | mmcblk0p3 | `/` | ext4 | (auto) | rootfs slot B, read-only, same size as A |
| p4 | mmcblk0p4 | `/data` | ext4 | 1G | persistent platform + app data (fixed size) |
| p5 | mmcblk0p5 | `/home` | ext4 | 500M | user home dirs, growfs to fill remaining eMMC |

**Note on partition numbering:** MBR with >4 partitions creates an extended partition container. p1-p3 are primary, p4 becomes the extended container, and logical partitions start at p5. So `/data` is `mmcblk0p5` and `/home` is `mmcblk0p6` at the block device level, even though they are the 4th and 5th entries in the WKS file. The fstab must use the actual device names (p5, p6), not the WKS entry order.

**Changes:**
- Create `layers/meta-appliance-bsp-reterminal/wic/appliance-dual-rootfs.wks.in` with the 5-partition layout.
- Set `WKS_FILE = "appliance-dual-rootfs.wks.in"` in kas `local_conf_header`.
- Set `IMAGE_FSTYPES:append = " ext4"` — RAUC bundles need raw ext4 rootfs images.

**Reasoning for 128M boot:** The FAT only holds firmware blobs + U-Boot + DTB overlays (kernel is in rootfs). 128M is generous; 64M would suffice but 128M leaves room for dual U-Boot in the future.

**Reasoning for auto-sized rootfs:** No fixed size. WIC's `--source rootfs` auto-sizes to fit the image content + overhead factor. Both A and B will be identical size. The current image is ~300MB; with overhead this will be ~400MB per slot.

**Reasoning for separate /home with growfs:** Matches upstream meta-rauc-raspberrypi exactly. `/data` has a fixed 1G allocation for platform and app state. `/home` gets the remaining eMMC space and grows on first boot via `x-systemd.growfs`. This separation keeps app data predictably sized while giving user home dirs room to grow.

### Step 4: Update fstab for A/B + read-only rootfs

The current BSP layer fstab mounts rootfs rw and doesn't mount `/data` or `/home`. It needs to:
- Mount rootfs read-only (`ro`)
- Mount `/data` from p5 (logical partition in extended container)
- Mount `/home` from p6 with `x-systemd.growfs`
- Keep tmpfs for volatile state

**Changes:**
- Replace `layers/meta-appliance-bsp-reterminal/recipes-core/base-files/files/fstab` with:
  ```
  /dev/root            /                    auto       ro                    1  1
  proc                 /proc                proc       defaults              0  0
  devpts               /dev/pts             devpts     mode=0620,ptmxmode=0666,gid=5      0  0
  tmpfs                /run                 tmpfs      mode=0755,nodev,nosuid,strictatime 0  0
  tmpfs                /var/volatile        tmpfs      defaults              0  0
  /dev/mmcblk0p1       /boot                vfat       defaults,nofail       0  0
  /dev/mmcblk0p5       /data                ext4       defaults,nofail       0  0
  /dev/mmcblk0p6       /home                ext4       x-systemd.growfs      0  0
  ```

**Note on dirs755:** The upstream meta-rauc-raspberrypi `base-files_%.bbappend` (priority 6) adds `dirs755 += "/data"`. Our BSP layer (priority 10) overrides `FILESEXTRAPATHS` for the fstab but does not interfere with `dirs755` — the additive `+=` from the upstream bbappend still applies. `/home` already exists in the rootfs by default. No additional `dirs755` needed.

### Step 5: RAUC system.conf

The meta-rauc-community layer provides a `system.conf` that maps slot A → p2, slot B → p3, bootloader=uboot, data-directory=/data/. This matches our layout exactly.

**Changes needed:** None — the upstream `system.conf` works as-is. The CA cert is the dev cert from meta-rauc-community for now.

If we need to customize `system.conf` later (e.g., add pre/post-install hooks), we'll add a higher-priority bbappend in our BSP layer.

### Step 6: RAUC signing keys

The meta-rauc-community layer ships development signing keys (`development-1.key.pem` / `development-1.cert.pem`). These are fine for dev/test. The `update-bundle.bb` recipe references them via `RAUC_KEY_FILE` and `RAUC_CERT_FILE`.

**Changes:** None for now. Before shipping to real users, generate a production keypair and override `RAUC_KEY_FILE`/`RAUC_CERT_FILE` in a custom bundle recipe.

### Step 7: Kernel in rootfs (not FAT)

The meta-rauc-community layer's `core-image-minimal.bbappend` handles this:
```
IMAGE_INSTALL:append = " kernel-image kernel-modules"
RPI_EXTRA_IMAGE_BOOT_FILES:remove = "${KERNEL_IMAGETYPE}"
```

This installs the kernel into the rootfs and removes it from the FAT boot partition. U-Boot's `boot.cmd.in` then loads `boot/Image` from the selected rootfs partition.

**Changes:** None — the upstream bbappend does the right thing. Verify at test time that `/boot/Image` is inside the rootfs ext4 and not on the FAT partition.

### Step 8: U-Boot boot script

The meta-rauc-community layer's `boot.cmd.in` implements A/B slot selection:
1. Reads `BOOT_ORDER` (default "A B"), `BOOT_A_LEFT` / `BOOT_B_LEFT` (default 3 each)
2. Iterates slots in order, decrements the chosen slot's counter, sets `bootpart` and `raucslot`
3. Appends `root=${bootpart} rauc.slot=${raucslot}` to kernel cmdline
4. Saves env, loads kernel from the chosen rootfs partition, boots

**Changes needed:** None. The upstream `boot.cmd.in` loads only the kernel (`boot/@@KERNEL_IMAGETYPE@@`). The DTB is already loaded by the RPi firmware at `${fdt_addr}` and passed through by U-Boot. DTBs living inside the rootfs is a future enhancement — for now the upstream behavior (DTBs on FAT, kernel in rootfs) is the tested path.

### Step 9: cmdline.txt

The current `CMDLINE` has `root=/dev/mmcblk0p2` set by meta-raspberrypi. The meta-rauc-community layer's `linux-raspberrypi_%.bbappend` removes it with `CMDLINE:remove:rauc-integration = "root=/dev/mmcblk0p2"`. U-Boot sets `root=` per slot at boot time.

**Changes:**
- Verify that `CMDLINE` does not contain a hardcoded `root=` after the rauc-integration override applies.
- Ensure `rootwait` is still present (required for eMMC boot per AGENTS.md gotcha).
- Add `ro` to cmdline to reinforce read-only rootfs (belt-and-suspenders with fstab `ro`).

Add in BSP layer.conf:
```
CMDLINE:append = " ro"
```

### Step 10: Grow data partition service

The upstream meta-rauc-raspberrypi `rauc-grow-data-partition.service` resizes both partition 4 (extended → data) and partition 6 (/home) to 100%. It is triggered `Before=home.mount` and `WantedBy=home.mount`. This matches our 5-partition layout exactly.

The upstream `rauc_%.bbappend` packages it as `rauc-grow-data-part` with dependencies on `parted`. Note: the upstream service does **not** include `resize2fs` — it relies on `x-systemd.growfs` in the fstab to handle filesystem expansion after the partition is resized.

**Changes:**
- Add `rauc-grow-data-part` to `IMAGE_INSTALL` in kas `local_conf_header` so the grow service is included in the image.
- No need to override the service — the upstream version handles our layout correctly.

### Step 11: Mark-good service

RAUC's slot-switching is "good path" only — after booting into the new slot, something must call `rauc status mark-good` to confirm the update succeeded. If this never happens (boot loops, critical service fails), U-Boot exhausts `BOOT_<X>_LEFT` and rolls back.

**Changes:**
- Create `recipes-core/rauc/files/appliance-mark-good.service`:
  ```ini
  [Unit]
  Description=Mark RAUC slot as good after successful boot
  After=multi-user.target
  ConditionPathExists=!/run/appliance-marked-good

  [Service]
  Type=oneshot
  ExecStart=/usr/bin/rauc status mark-good
  ExecStartPost=/usr/bin/touch /run/appliance-marked-good

  [Install]
  WantedBy=multi-user.target
  ```
- Create `appliance-mark-good_1.0.bb` recipe in `meta-appliance-os` to install the service.
- Add `appliance-mark-good` to `IMAGE_INSTALL` in kas `local_conf_header`.
- For now, trigger after `multi-user.target` (all services up). Later phases will refine this to depend on specific critical services (kiosk-shell, kiosk-buttond, kiosk-playd).

### Step 12: Update bundle recipe

The meta-rauc-community layer's `update-bundle.bb` builds a RAUC bundle for `core-image-minimal`. This should work as-is since that's our current image recipe. The bundle includes an ext4 rootfs image and is signed with the dev key.

The upstream recipe hardcodes `RAUC_BUNDLE_VERSION = "v20200703"`. We should override this.

**Changes:**
- Create a `update-bundle.bbappend` in our OS layer to set `RAUC_BUNDLE_VERSION` and `RAUC_BUNDLE_DESCRIPTION` dynamically from `DISTRO_VERSION`.
- When we move to `appliance-os-image`, also update `RAUC_SLOT_rootfs` in the bbappend.

### Step 13: Build and test

**Build verification:**
1. `make build` produces a `.wic.bz2` with 5 partitions (verify with `fdisk -l`)
2. `bitbake update-bundle` produces a `.raucb` file
3. Flash `.wic.bz2` to eMMC via rpiboot
4. Device boots into slot A via U-Boot
5. `/etc/os-release` shows correct version
6. `rauc status` shows slot A active/booted, slot B inactive
7. `mount` shows `/` as `ro`
8. `/data` is mounted rw
9. `/home` is mounted rw and grows to fill remaining eMMC
10. Install bundle to slot B: `rauc install /path/to/bundle.raucb`
11. Reboot → boots into slot B
12. `rauc status mark-good` runs → slot B marked good
13. Simulate failure: break a critical service on slot B, reboot 3 times → falls back to slot A
