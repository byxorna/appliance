# App Container Environment File

## Intent

Let users inject environment variables into appliance app containers by editing a plain text file on the persistent partition. No `systemctl edit`, no rebuild, no bbclass knowledge needed. Just edit the file and restart the service.

## Detailed Implementation Plan

Each app gets `/data/apps/<name>/env` as a podman `--env-file`. The file uses standard `KEY=VALUE` format (podman's native env-file format, one per line, `#` comments allowed).

Changes to `appliance-app.bbclass`:

1. Add `--env-file /data/apps/<name>/env` to the generated `podman run` command, after the hardcoded `-e` flags. This means the env file can override built-in env vars if needed.
2. Add an `ExecStartPre` line that touches the env file if it doesn't exist, so podman doesn't fail on a missing `--env-file`. Same pattern already used for the pipewire socket touch.
3. The tmpfiles.d conf already creates `/data/apps/<name>/` with correct ownership. The env file gets created empty by ExecStartPre on first run, owned by root (acceptable since podman reads it, not the container process).

No changes to app.json schema. No changes to individual app recipes.

## Reasoning

`--env-file` is the right primitive here. It's native to podman, well-documented, supports comments, and avoids the `EnvironmentFile=` + `Environment=` + shell quoting dance that systemd would require to thread values through to `podman run -e`. The env file lives on `/data/` so it survives A/B rootfs updates. Placing it inside the app's existing data dir (`/data/apps/<name>/`) keeps the filesystem tidy and means the existing tmpfiles.d ownership covers the parent directory.

Putting `--env-file` after the hardcoded `-e` flags means user values win on conflict (podman last-write-wins).
